# 見出し・サマリー・目次を持つ長文の生成

> **⚠️ このプランは [docs/plans/topic-variants-and-length-tiers.md](topic-variants-and-length-tiers.md) に統合・置き換えられた。**
> トピックが複数バリアント（レベル×分量）を持つ設計に変わったことに伴い、`config.json` の構造・`workflows/config.md` /
> `workflows/outline.md` の役割分担が新プランで再設計されている。以降の実装作業は新プランを参照すること。
> Phase 1（`esl-level-spec.md` の長文モード概念・`personas/esl-writer.md` の longForm 出力フォーマット）で作った内容は
> 新プランに引き継がれるが、`config.md` / `outline.md` の該当箇所は新プランのPhase Bで作り直す。

## 目的・背景

現状の生成物は「H1タイトル＋段落の羅列」のみの単一ブロック構成（例: `texts/water-cycle-*/articles/v1.md`）。
語数（テキストの分量）が大きくなると、単一ブロックのままでは読みにくく、全体像も掴みにくい。

**レベル（語彙・文法の難度）と語数（分量）は本来独立したパラメータ**であり、両者を混同しない設計にする。
たとえば「A1の易しい語彙のまま語数だけ増やした長い物語」も「C1の難しい語彙のまま語数を絞った短い文章」もあり得る。
そのため長文構成（見出し・サマリー・目次）を使うかどうかは**レベルではなく確定した語数**で判定する
（詳細は下記「1. 適用条件」）。

`~/deep-pulse` の記事は「キャッチーな見出し＋リード文（サマリー）＋（長い記事では）セクション見出し＋自動生成の目次」という
構成を取っており、読者が長文でも迷わず読めるようになっている。ESL学習者向けにも同様の構成を導入し、
長文でも「今何を読んでいるか」「全体で何を扱うか」が把握しやすい教材にする。

deep-pulse の「多階層記事（chapters/ でページ分割）」までは今回のスコープに含めない。今回追加するのは
**単一ページ内でのリード文（サマリー）＋セクション見出し＋目次**まで。ページ分割が必要になるほど長い記事は
将来必要になった時点で別タスクとして検討する。

## 対応方針

### 1. 適用条件（長文モード）

- `config.json` に `longForm: boolean` を追加する
- `workflows/config.md` で語数目安を確定する際、レベルのデフォルト語数に関わらず**利用者が自由に語数を上書きできる**ことを明示する
  （レベルは語彙・文法の難度のみを決め、語数はレベルとは別に確定する）
- 語数目安確認の直後に、**確定した `wordCountTarget` の下限が概ね600語以上の場合のみ**「長文構成（見出し＋サマリー＋セクション分け）にするか」を確認する設問を追加する
  - レベルは問わない（A1でも語数が600語以上なら対象になる）
  - 600語未満の場合は単一ブロックの短い文章がESL読解教材として適切なため、長文モードの設問自体を出さず `longForm: false` 固定とする
  - 未指定時のデフォルトは `false`（従来どおりの単一ブロック）

### 2. アウトライン（`workflows/outline.md`）

- `longForm: true` の場合、各セクションに「本文で使うH2見出し文言（英語、指定レベルの語彙に収まる短いフレーズ）」を追記させる
- `longForm: false` の場合は現行どおり（セクションは構成メモであり、本文には見出しとして現れない）

### 3. 本文生成（`workflows/generate.md` / `personas/esl-writer.md`）

`longForm: true` の場合の出力形式を以下に固定する（`personas/esl-writer.md`の出力形式節に追記）。

```markdown
# タイトル

（リード文: 1段落、記事全体が何を扱うかを短く要約する。指定レベルの語彙・文長を守る。装飾や煽り文句よりも「何が読めるか」が分かることを優先する）

## セクション見出し1

本文…

## セクション見出し2

本文…
```

- 目次は**本文には書かない**（下記の表示側で自動生成するため、手書きの目次・アンカーリンクを本文に含めない）
- リード文は既存の執筆前チェックリスト（語彙・文長・レベル適合）の対象に含める
- `longForm: false` の場合は現行の出力形式（H1＋段落のみ）を維持し、変更しない

### 4. 表示側の目次自動生成（`lib/site.js`）

手書きの目次・アンカーだと表記ゆれで壊れやすいため、deep-pulse と同様に**見出しから自動生成**する。

- `marked` のカスタムレンダラーで H2 見出しに連番ベースの `id`（`heading-0`, `heading-1`, ...）を自動付与する
  - 日本語スラグ化はしない（deep-pulseと同じ理由: ライブラリ依存で挙動がブレるため）
- 本文中に H2 が1つ以上あれば、リード文（最初の段落）の直後・最初のH2の直前に目次ブロックを自動挿入する
- H2 が0個の記事（従来の短文記事）は目次を生成しない＝**見た目・挙動とも一切変化なし**
- 対象は記事本文（`renderArticle`）と一覧詳細ページのプレビュー（`renderTextDetail` 内の本文プレビュー）の両方
- 実装は `lib/site.js` に集約されているため、`server.js`（動的表示）・`scripts/build-static-site.js`（静的ビルド）の両方に自動的に反映される（個別の変更は不要）
- 既存の Playful デザイン（`.badge` の丸みや配色）に合わせた目次用CSSを `layout()` 内に追加する

## 影響範囲

- `docs/specs/esl-level-spec.md` — 長文モードの適用条件・構成要素の説明を追記
- `workflows/config.md` — 長文モード確認の設問追加、`config.json` に `longForm` 追加
- `workflows/outline.md` — `longForm: true` 時のセクション見出し文言の記録
- `workflows/generate.md` / `personas/esl-writer.md` — `longForm: true` 時の出力形式（リード文＋H2セクション）
- `lib/site.js` — 見出しへのid付与、目次自動生成・挿入ロジック、CSS追加
- `workflows/brushup.md` — 変更なし（既存の構成変更／本文調整のフローがそのまま使える想定。Phase 3の実地検証で問題があれば追記する）
